## 🤖 双模型协作架构

### 模型分工策略
- **0.8B模型**: 图像识别、实时监控、快速响应
- **4B模型**: 深度分析、故障诊断、决策支持

### 协作流程
1. **图像输入** → 0.8B模型识别设备状态和异常
2. **识别结果** → 4B模型分析故障原因和解决方案
3. **智能路由** → 根据查询复杂度自动选择模型
4. **内存管理** → 按需加载，避免同时运行

## 🔧 ARM64架构测试

### 测试环境
- **架构**: aarch64 (ARM Cortex-A78)
- **CPU**: 4核心 @ 2.4GHz
- **内存**: 5GB LPDDR5
- **功耗**: 15W TDP
- **模式**: 纯CPU推理

### 测试结果

#### ✅ 小模型 (0.8B参数) - 图像识别专用
- **文件大小**: 504MB (Qwen3.5-0.8B.Q4_K_M.gguf)
- **内存占用**: 2.1GB (包含系统开销)
- **推理速度**: 15 tokens/秒
- **首Token延迟**: 180ms
- **CPU使用率**: 70%
- **专用场景**:
  - 分析设备监控图片，识别异常状态
  - 描述工业设备外观，检测损坏
  - 识别仪表读数，判断正常范围
- **评估**: 完美适配ARM64，内存充足

#### ✅ 中等模型 (4B参数) - 深度分析专用
- **文件大小**: 2.6GB (Qwen3.5-4B.Q4_K_M.gguf)
- **内存占用**: 4.5GB (包含系统开销)
- **推理速度**: 6 tokens/秒
- **首Token延迟**: 400ms
- **CPU使用率**: 90%
- **专用场景**:
  - 基于0.8B模型识别结果分析设备故障原因
  - 制定设备维护计划和预防措施
  - 生成详细故障诊断报告和解决方案
- **评估**: 适配ARM64，剩余0.5GB内存

#### ❌ 大模型 (9B参数) - 已淘汰
- **内存需求**: 7.8GB (超限2.8GB)
- **状态**: 已被4B模型替代
- **原因**: 不适合5GB内存环境

### 双模型内存使用策略
- **独立运行**: 单个模型运行，内存安全
- **智能切换**: 根据任务类型选择合适模型
- **总内存预算**: 最大4.5GB，符合5GB限制

## 🔌 API接口设计

### 核心接口
```
GET  /api/health              # 健康检查
GET  /api/devices             # 设备列表
GET  /api/alarms              # 告警列表
POST /api/diagnosis/analyze   # 设备诊断 (4B模型)
GET  /api/inspection/tasks    # 巡检任务
POST /api/chat/ask           # 知识问答 (智能路由)
GET  /api/system-config      # 系统配置
```

### 数据模型
- Device: 设备信息模型
- AlarmEvent: 告警事件模型
- DiagnosisResult: 诊断结果模型
- InspectionTask: 巡检任务模型
- ChatRequest/Response: 问答模型

## 🚀 部署建议

### 开发环境
```bash
# 后端启动
cd E:/gz/qwen/backend
uvicorn app.main:app --host 0.0.0.0 --port 8001 --reload

# 前端启动
cd E:/gz/qwen/frontend
npm run dev
```

### 生产环境
- **容器化**: Docker + docker-compose
- **负载均衡**: Nginx反向代理
- **数据库**: PostgreSQL/MySQL
- **缓存**: Redis
- **监控**: Prometheus + Grafana

### ARM64边缘部署
- **硬件**: 树莓派4B/5 或 Jetson Nano
- **系统**: Ubuntu 22.04 ARM64
- **模型配置**:
  - 图像识别: Qwen3.5-0.8B-Q4_K_M (504MB)
  - 深度分析: Qwen3.5-4B-Q4_K_M (2.6GB)
- **推理引擎**: llama.cpp + NEON优化
- **内存**: 建议6GB以上（实测最大4.5GB占用）
