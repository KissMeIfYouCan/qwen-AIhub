# AI 中控 vibe coding 整体方案

> 面向“传统物联网主控 + AI 中控”分层融合架构的实施型方案文档。目标不是做一份汇报材料，而是沉淀一份可以直接继续交给 AI 生成 PRD、页面、组件、接口、数据库设计和原型的基础文档。

---

## 1. 项目定位

### 1.1 项目名称
AI 中控智慧监测与运维协同平台

### 1.2 项目一句话定义
以传统物联网主控作为稳定运行底盘，以 AI 中控作为智能分析与辅助决策中枢，形成一个可监测、可诊断、可问答、可生成报告、可继续由 AI 自动拆解开发的智能运维平台。

### 1.3 项目核心目标
1. 保留传统物联网系统在采集、组网、控制、告警上的稳定性。
2. 将 AI 能力集中在异常诊断、巡检报告、视觉识别、知识问答、Skill 调度、RAG 增强等高价值环节。
3. 对外提供统一的 Web 管理后台、驾驶舱大屏、移动巡检端，形成“展示合一、底层解耦”的产品形态。
4. 让整套方案适合 vibe coding：后续可以直接让 AI 按模块生成页面、接口、数据结构、原型和任务分解。

---

## 2. 总体架构原则

### 2.1 架构结论
本方案采用：

**传统物联网控制层 + AI 智能分析层** 的双层协同架构。

### 2.2 分层边界

#### 传统物联网主控负责
- 传感器接入与实时采集
- Zigbee / LoRa / RS485 / Modbus 等通信组网
- 网关接入与协议转换
- 云平台或本地平台上报
- 告警触发
- 联动控制
- 基础监控与设备状态展示

#### AI 中控负责
- 异常诊断分析
- 巡检报告生成
- 图像 / 热成像 / 多模态识别
- 知识问答
- 运维助手
- Skill 工作流调度
- RAG 检索增强
- 结构化结论回写到前端界面

### 2.3 架构设计原则
- **控制与智能分离**：AI 不直接接管底层实时控制。
- **系统解耦**：AI 故障不影响主控基础运行。
- **展示合一**：用户看到的是一个统一产品，而不是两套割裂系统。
- **面向实施**：每个模块都能进一步拆为页面、接口、表结构和 AI 开发任务。
- **渐进升级**：先接入数据，再做诊断，再做报告，再做问答与自动化流程。

## 3. 目标用户与角色

### 3.1 平台管理员
关注点：系统配置、设备接入、权限分配、模型配置、知识库维护、日志审计。

### 3.2 运维人员 / 巡检人员
关注点：告警处理、异常定位、现场巡检、拍照上传、报告生成、维修建议。

### 3.3 管理者 / 负责人
关注点：驾驶舱总览、风险分布、设备健康度、巡检完成率、异常闭环率。

### 3.4 培训 / 演示人员
关注点：知识问答、答辩辅助、案例检索、方案展示、系统讲解。

## 4. 核心使用场景

### 场景 1：实时监测与告警联动
传统物联网主控持续采集温湿度、水位、火焰、风速风向、加速度等数据；达到阈值后触发告警，并同步把事件推送给 AI 中控进行原因分析。

### 场景 2：异常诊断
AI 中控读取告警数据、设备状态、日志、历史案例，生成：
- 可能原因
- 排查优先级
- 处理建议
- 风险等级

### 场景 3：智能巡检
运维人员通过移动端上传现场图片、热成像、定位信息和备注；AI 自动生成结构化巡检结论和巡检报告。

### 场景 4：知识问答
用户输入“某设备频繁离线的原因是什么”“火焰传感器误报怎么排查”等问题，系统通过知识库和案例库输出可执行答案。

### 场景 5：管理驾驶舱展示
在大屏或后台首页统一展示实时数据、告警分布、设备状态、AI 分析摘要、巡检完成情况。

### 场景 6：答辩 / 演示辅助
系统可将项目方案、评分标准、设备说明书、历史讲稿纳入知识库，用于答辩问答和方案说明。

- 常见故障案例
- 讲稿与培训资料

## 9. 数据流与系统边界

## 9.1 主数据流
```text
传感器 / 摄像头 / 热成像设备
  ↓
传统物联网主控
  ↓
实时数据、告警事件、设备状态
  ↓
AI 中控服务层
  ↓
异常诊断 / 报告生成 / 问答 / 视觉分析
  ↓
统一回写到 Web / 大屏 / 移动端
```

## 9.2 系统边界定义

### 主控边界
只负责“稳定运行必需能力”，包括采集、组网、控制、告警。

### AI 边界
只负责“智能分析与辅助决策能力”，包括诊断、报告、识别、问答、知识增强。

### 前端边界
对用户暴露统一产品体验，不要求用户理解两套后台的物理分离。

## 11. 技术实现建议

## 11.1 前端建议
适合 vibe coding 的推荐技术栈：
- Web 管理后台：React + TypeScript + Ant Design / Arco Design
- 大屏：React + ECharts
- 移动端：uni-app / React Native / H5
- 状态管理：Zustand 或 Redux Toolkit
- 数据请求：Axios / TanStack Query

前端页面生成建议：
- 先生成页面骨架
- 再生成组件
- 再接 API mock
- 最后替换真实接口

## 11.2 后端建议
- API 服务：Node.js NestJS / Java Spring Boot / Python FastAPI 均可
- 推荐优先：**FastAPI 或 NestJS**，更适合快速迭代和 AI 协同开发
- 任务队列：Celery / BullMQ
- 文件存储：MinIO / 本地对象存储
- 数据库：PostgreSQL / MySQL
- 缓存：Redis

## 11.3 AI 服务建议
- LLM 服务：统一封装大模型调用层
- Prompt 模板中心：管理诊断、报告、问答模板
- RAG 服务：文档切片、向量化、检索、引用返回
- 多模态分析服务：图像、热成像输入解析
- Skill 编排服务：把复杂任务转为多步骤可执行链路

## 11.4 RAG 建设建议
知识库优先级：
1. 设备说明书
2. 常见故障案例
3. 项目架构文档
4. 巡检模板与报告样例
5. 评分标准 / 讲稿 / 培训资料

RAG 输出要求：
- 必须返回引用来源
- 必须区分“知识库答案”和“模型推断”
- 对高风险建议增加人工复核提示

## 11.5 接口设计建议
建议按域拆分 API：
- `/api/devices`
- `/api/telemetry`
- `/api/alarms`
- `/api/diagnosis`
- `/api/inspection`
- `/api/reports`
- `/api/knowledge`
- `/api/chat`
- `/api/skills`
- `/api/system-config`

## 12. 面向 vibe coding 的开发拆解方式

## 12.1 推荐开发顺序
### 第一阶段：搭框架
- 先搭后台框架和菜单结构
- 先定义基础路由和页面占位

### 第二阶段：接静态原型
- 先做驾驶舱、诊断页、巡检页、问答页、配置页

- 接入诊断生成
- 接入问答
- 接入报告生成
- 接入知识库引用

### 第五阶段：优化闭环
- 增加处理记录
- 增加导出能力
- 增加权限控制
- 增加日志审计

## 12.2 适合继续交给 AI 的任务类型
- 生成页面原型
- 生成 React / Vue 页面代码
- 生成 API 定义
- 生成数据库表结构
- 生成 Prompt 模板
- 生成测试用 mock 数据

## 15. 给 AI 继续开发时的统一约束

后续让 AI 生成页面、模块、接口时，建议统一附加以下约束：
- 保持“传统主控稳定底盘 + AI 中控智能增强”的架构边界，不要把 AI 写成直接控制设备。
- 页面风格偏工业监测 / 智慧运维 / 管理后台，不要做成消费级产品风格。
- 所有诊断结论、报告结论、问答结果都要预留“来源依据 / 置信度 / 人工复核”区域。
- 模块设计优先可拆分、可复用、便于继续生成。
- 前端优先组件化结构，后端优先按领域拆 API。
- 默认支持假数据演示，再逐步替换真实接口。

## 16.1 管理后台首页 / 驾驶舱提示词

```text
请为“AI 中控智慧监测与运维协同平台”生成一个 Web 管理后台首页 / 驾驶舱原型方案。

背景：
```

## 17. 整体方案正文结论

本方案不是单纯增加一个“AI 功能模块”，而是把 AI 中控定义为传统物联网主控之上的智能分析与协同中枢。

整体实施思路如下：

1. **底层稳定运行不动摇**：继续由传统物联网主控负责传感器采集、设备接入、协议组网、网关上云、告警触发、联动控制，确保系统实时性与稳定性。
2. **上层智能能力集中建设**：由 AI 中控统一承接异常诊断、巡检报告、视觉识别、知识问答、Skill 调度、RAG 增强等能力，避免把 AI 强行塞进底层控制链路。
3. **前端统一呈现**：通过 Web 管理后台、驾驶舱大屏、移动巡检端把实时监测、AI 诊断、巡检闭环、知识问答整合为一个统一产品体验。
4. **开发方式面向 vibe coding**：先定义页面边界，再定义组件边界，再定义接口与数据结构，最后接入 AI 服务和知识库，让 AI 可以持续参与设计、生成、迭代。
5. **系统演进路径清晰**：先做可演示 MVP，再做业务闭环，再做多模态与多 Skill 增强，最终形成一个可部署、可扩展、可维护的 AI 中控平台。

因此，这份文档可以直接作为后续生成 PRD、页面原型、前端代码、后端接口、数据库设计、Prompt 模板和 Skill 工作流的母文档。


如果继续 vibe coding，建议按以下顺序继续生成：
1. 先生成整套后台的信息架构和菜单路由。
2. 再生成首页 / 驾驶舱静态页面。
3. 再生成异常诊断页和巡检报告页。
4. 再生成知识问答页和系统配置页。
5. 再生成后端 API 定义与 mock 数据。
6. 最后补充数据库设计与 Prompt / Skill 配置。

这样可以最快形成一个可演示、可扩展、可继续自动生成的 AI 中控平台雏形。
